# 社媒内容自动化分析与筛选系统架构设计

本文档规划了该自动化项目的整体架构、模块职责、日志规范、分模块调试机制以及各核心模块间的接口设计。后续编码需严格遵从该骨架，以确保系统的高内聚、低耦合与高可维护性。

---

## 1. 系统整体架构 (System Architecture)

系统根据时间的调度与数据的流向可以清晰分为两大部分：**数据处理流水线 (Pipeline)** 和 **前端展示与调试站 (Web UI)**。为保证项目可以在断电/异常后恢复与重试，各模块间**禁止内存层次的硬传递**，所有模块间的信息通讯均通过**本地数据库（如 SQLite）或具有标准结构的 JSON / 文件系统**进行流转。

系统的数据生命周期如下：

1. **调度模块 (Scheduler)**：配置整个自动化流程的触发时间、频率和全局参数。
2. **采存一体化爬虫模块 (Integrated Crawler & Filter Module)**：
   - 核心基于开源的 MediaCrawler 项目魔改。
   - **抓取与初筛联通**：在获取出最新帖子/视频的基础元数据后，立刻进行内置的**初筛逻辑（依据标题、标签、博主信息等）**。
   - **媒体按需下载**：对于经过初筛判定合格的内容，顺势复用爬虫上下文的请求态完成高清图片、视频的下载（内部保留「视频文件过大拦截」机制）。不合格的直接丢弃媒体下载过程。
3. **视频截图模块 (Video Screencapper)**：对成功下载到本地的视频按照既定规则抽帧截取画面，输出图片给下一步。
4. **OCR提取模块 (OCR Processor)**：利用 PaddleOCR 对所有小红书的配图与B站视频抽帧截图进行文字识别。（需支持环境自适应：PC启动GPU加速，MAC默认使用CPU）。
5. **结果展示与实时筛选模块 (Dashboard Querying Web)**：面向最终用户的前端页面，直接基于数据库中的 `title + desc + ocr_text_joined` 做实时筛选并展示命中结果。
6. **专属调试网站 (Debug Web)**：面向开发与维护人员的前端面板，用于单步执行模块、配置管理和错误排查。

## 2. 规范化日志 (Standardized Logging) 

为了能够在异步环境和拆解的模块中顺畅追溯问题，项目将采用一套严谨的「格式化日志输出规范」。

### 2.1 日志层级设计
- **INFO**：记录正常流水线的节点级变更，如 `[OCR模块] 解析完成条目 ID: xxx`。
- **WARNING**：不影响整体流程，但产生资源丢弃的事件。例如视频过大被抛弃、单幅图片下载失败。
- **ERROR**：影响某功能模块正常运行的关键性失败，如无网络连接、数据库写入锁定、PaddleOCR 模型未找到等。
- **DEBUG**：开发态独有的详尽输出。例如发送爬虫的 HTTP 响应体片段或某条 SQL 的详细 Query。

### 2.2 数据格式与 Trace_ID 切片
每运行一次定时自动化任务，系统都会在最开始生成一个唯一的字符串 **`Task_ID` (Trace ID)**。
所有的日志记录都必须遵守以下 JSON 格式或等效的字符串模板规范：
`[Timestamp] [Task_ID] [Module_Name] [LEVEL] - Message | JSON_ExtraData`

- 落盘策略：每日轮转 (Daily Rotating) 日志文件保存在本地 `logs/` 文件夹下。
- 支持针对单一 `Task_ID` 的日志快速提取功能。

## 3. 分模块开发与调试机制 (Modular Debugging Mechanism)

整个项目的核心难点在于不同模块的依赖问题。为了实现「能够有一个专门的调试用网站用于便捷地调试每一个模块」，需遵循以下开发规范：

### 3.1 极简依赖与解耦
模块仅仅需要知道两件事：1. 我要去哪里读什么数据；2. 我要把结果写到哪里。
- 提供一致的执行入口：所有的模块需要封装成对象或包含一个 `run(task_id=None, item_ids=[])` 的主要运行函数。

### 3.2 调试 Web 框架的设计
专属调试站点（建议采用 FastAPI 或 Flask 轻量级开发）直接接驳各模块核心类。
1. **测试数据 Mock**：在调试平台，可以不运行上游（如跳过下载），直接上传几张图作为 Mock 数据塞到当前待调试的 `Item_ID` 身上，点选运行某模块（如只对该照片执行 OCR）。
2. **级别动态提权**：在网页端点击“调试”时，后端可临时将该模块运行进程的 Log Level 改为 `DEBUG`，并将日志通过 `WebSocket` 或流式接口实时推送到前端网页显示。
3. **隔离防污染**：在调试模式下，如果使用真实的单条数据进行模块试运行，应通过设置 `dry_run=True` 阻断其向主流程 SQLite 覆盖写回结果，或将其改写至临时缓存表进行验证。

## 4. 模块间的接口设计 (Inter-Module Interface Specification)

系统中所有流转数据的“承载容器”是一个标准的抽象模型，假定为 `MediaItem`。该模型记录着这一条内容从“刚刚爬取”到“最终展示”所叠加出现的所有数据。

各模块的输入输出契约规定如下：

### 4.1 数据载体 `MediaItem` (基础 Schema 概念)
```json
{
  "item_id": "bilibili_BV1xx/xhs_id", // 唯一标识符
  "task_id": "20240501-A1B2",          // 本次抓取的批次
  "platform": "bilibili | xhs",
  "base_info": {                       // 爬虫入库的原始字段
    "title": "标题",
    "desc": "内容描述文字",
    "publish_time": "..."
  },
  "initial_passed": true,              // 首次过滤结论
  "local_media": {
    "video_path": "/local/data/xxx.mp4",
    "image_paths": ["/local/data/img1.jpg"]
  },
  "video_screenshots": ["/local/data/shot1.jpg"], // 视频截图路径
  "ocr_text_joined": "图1有猫 图2有狗",             // OCR提取结果合计（已落库）
  "display_match": {                              // 展示层实时计算（当前不落库）
    "passed": true,
    "matched_keywords": ["猫", "狗"],
    "matched_by": ["title", "ocr_text_joined"]    // 命中字段来源
  }
}
```

### 4.2 具体模块输入输出规划

**(1) 采存一体化爬虫模块 Integrated Crawler Module**
- **Input (启动参数)**: `(task_id: str, platform_config: dict, initial_filter_rules: dict, max_video_size_mb: int)`
- **Output**: 
  - 批量生成完整的 `MediaItem` 写入数据库。
  - 内部机制：调用 MediaCrawler，拿到页面信息后输入到初筛逻辑校验。
  - 若初筛通过，变更 `initial_passed=True`，并使用爬虫本身的下载功能将媒体落盘，地址存入 `local_media`。且依然要进行 Header `Content-Length` 的防超限阻断。
  - 若不通过，变更 `initial_passed=False`，抛弃媒体下载步骤，依然保留一条基础信息骨架写入数据库以供排查。
- **当前实现状态**：时间窗口过滤与视频大小拦截已完成。标题/标签/博主等初筛规则已预留扩展接口（`tools/filter_rules.py → ItemFilter`），按需新增 `@staticmethod` 方法即可。

**(4) 视频截图模块 Video Screenshot Module**
- **Input**: `(item_ids_to_screenshot: List[str], screenshot_rules: dict)`
- **Output**: 将生成的图片存放至抽帧专属目录，将图片绝对路径集合写入数据库对应字段 `video_screenshots`。

**(5) OCR处理模块 OCR Processor Module**
- **Input**: `(item_ids_to_ocr: List[str])`
- **Output**: 按 `device=auto/cpu/gpu` 策略自动选择设备（当前约定：macOS 下 `auto` 默认 CPU），解析该 ID 下拥有的所有图片（涵盖原图及截图），拼接 OCR 得出的文字存入 `ocr_text_joined`。

**(6) 展示检索一体模块 Dashboard Querying Module**
- **Input**: `(task_id: str | None, query_keywords: List[str], time_range: tuple | None, platform: str | None, limit: int)`
- **Output**: 对完成 OCR 的条目，以 `base_info.title + base_info.desc + ocr_text_joined` 作为联合检索目标，实时返回命中列表与命中解释（`matched_keywords` / `matched_by` / `score`）。默认不写回数据库、不打持久化标签。

**(7) 调试平台 Debug Web**
- 对接各个独立封装模块的 `run()` 函数，提供按钮形态 GUI 与日志追踪能力。
- 在页面上提供与展示检索一致的筛选参数输入，便于联调时复现实战查询条件。

### 4.3 展示检索与独立筛选的拆分边界

当前阶段采用“展示检索一体”方案（不落库打标签），以提升迭代速度并降低维护成本。后续当满足以下任一条件时，再拆分为独立筛选模块：
- 单批次数据量明显上升（如达到千级以上）导致实时检索延迟不可接受。
- 规则复杂度明显上升（多轮打分、模型推理或需要稳定版本化结果）。
- 业务需要对“当时命中结果”做审计回放，必须持久化筛选结论与命中证据。

## 5. 程序分发与依赖环境管理 (Distribution & Environment Management)

由于本项目模块众多，且未来集成的模块（如 OCR 服务）可能存在复杂的 C++ 或平台特定底层依赖。为了防止 **“在开发者电脑能跑，换台电脑就崩溃”** 的环境依赖灾难，本项目强制规定了以下“一键式绿色独立沙箱”的分发与运行机制：

### 5.1 禁绝全局与智能依赖查找
项目中的任何跨目录、子进程唤醒调用（如 `subprocess.run()` 启动爬虫或截图），**严禁写死解释器的物理绝对路径**，也**严禁仅使用 `"python"` 等依赖系统环境变量的模糊命令**。

系统中所有的程序调用必须通过**拼接基于当前根目录的相对执行路径**来实现，例如：
```python
current_env_python = os.path.join(project_root, "venv", "bin", "python")
subprocess.run([current_env_python, "xxx.py"])
```

### 5.2 统一的 `requirements_all.txt`
取消各子目录的“小算盘”。根目录下设有一个统一的 `requirements_all.txt` 文件。
爬虫需要的 `playwright`、截图需要的 `opencv`、文字提取需要的 `PaddleOCR` 均统一汇总在此，方便统一管控防冲突。

### 5.3 傻瓜式的「双击部署」与「双击运行」标准
交付给其他无开发经验的使用同事（运营、分析师）时，仅需把本文件夹打包（不含 venv）。
并提供放置于项目根目录的两把“双击钥匙” (`.command` 用于 Mac，`.bat` 用于 Win)：

- **首次运行依赖安装脚本**：双击后，该脚本会在当前位置拉起原生的 `python3 -m venv venv` 建立纯净沙箱，并全自动将其激活后安装大全清单。（此操作一劳永逸）
- **日常运行使用脚本**：双击启动，脚本必须内部声明死锁调用已生成的 `./venv/bin/python` 沙箱环境来进行调度。

采用这种 **Local Venv + Batch Wrapper** 模式可实现 100% 透明的代码审计，与 100% 解耦的环境锁定。当环境污染或崩溃时，用户仅需删除 `venv` 文件夹一键重装即可。
