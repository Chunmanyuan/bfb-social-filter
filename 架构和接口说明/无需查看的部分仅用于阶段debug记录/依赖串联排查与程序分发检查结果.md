# Python 多模块串联运行依赖孤立问题排查与分发说明

## 1. 发生了什么？
在根目录（`新项目/`）下运行 `测试模块连贯运行.py` 时，脚本试图通过 `subprocess.run()` 启动 `MediaCrawler/` 目录下的爬虫程序。
但爬虫程序一启动就报错：`ModuleNotFoundError: No module named 'typer'`。

然而，如果直接进入 `MediaCrawler/` 目录执行同样的爬虫代码，却一切正常，没有任何报错。

## 2. 为什么会这样？（原理简述）
现代 Python 项目为了防止不同项目的依赖包（如 A 项目要 requests 1.0，B 项目要 requests 2.0）打架，通常会使用**虚拟环境 (Virtual Environment)** 技术进行隔离。

在我们的项目中，`MediaCrawler` 已经通过工具（`uv` / `pyenv`）配置了自己专属的虚拟隔离环境，核心依赖（如 `typer`, `playwright` 等）全部安装在这个专属环境里，而不是安装在电脑全局。

### 当你在 `MediaCrawler` 目录直接运行时：
系统环境管理工具检测到了该目录下的标识文件（如 `.python-version`），所以**智能地帮我们**把运行命令导向了那个存有 `typer` 的专属 Python 解析器。

### 当你在外层目录（`新项目/`）运行串联测试脚本时：
测试脚本使用了类似 `subprocess.run(["python", "_temp_e2e_runner.py"])` 的命令去跨目录调用。
这里的陷阱在于：**子进程所启动的 `python`，继承的是外层（甚至是全局）的 Python 环境空间**。因为跨目录调用丢失了上下文，它并没有进入 `MediaCrawler` 的专属虚拟环境。跑到一半发现“没装 `typer`”，于是直接崩溃。

## 3. 我们尝试了什么，最终是如何解决的？

**尝试 1**：把 `sys.executable` 改成 `python` 甚至尝试了外层的 `/Users/pan/.pyenv/shims/python`。
*结果*：依然报错。因为 `pyenv shims` 在子进程调用的场景下，仍然无法正确代理到子目录的真实上下文。

**尝试 2（解法一）**：使用包管理器强制切环境，将命令改为 `subprocess.run(["uv", "run", "python", ...])`。
*结果*：触碰到了编译墙。由于 `uv` 会对环境做严格的安全检查，发现某些底层库（如 `Pillow` 的 jpeg 头文件）缺失时触发了由源码重新全量编译的动作而报错退出。

**最终解决（解法二）**：**绝对路径直指核心靶心**。
放弃所有智能代理和包装器，我们直接溯源查询到了 `MediaCrawler` 专属依赖包的**真正的、绝对的 Python 物理执行路径**（在本次环境中位于 `/Users/pan/.pyenv/versions/3.12.11/bin/python`）。
我们在 `测试模块连贯运行.py` 中写死了这个绝对路径：
```python
python_executable = "/Users/pan/.pyenv/versions/3.12.11/bin/python"
subprocess.run([python_executable, "_temp_e2e_runner.py"], cwd=crawler_dir, check=True)
```
这样一来，无论自动化脚本在哪里启动，它都会**揪住那个带有完整爬虫依赖资产的 Python 执行文件去干活**，问题迎刃而解。

---

## 4. 当前架构会影响程序分发吗？

**结论是：会有直接影响。**

目前代码中写死了绝对路径 `/Users/pan/.pyenv/versions/3.12.11/bin/python`，这带有极强的**个人电脑环境特征**。如果直接把这套代码发给团队其他同事，或者扔到云服务器上运行，必然会因为找不到这个绝对路径而报错。

### 如何优化分发架构？
如果未来要分发给其他机器运行，建议做如下改造以抹平环境差异：

1. **统一启动脚本 (Shell Wrapper)**：
   在根目录写一个 `startup.sh`，在里面先通过 `source MediaCrawler/.venv/bin/activate`（或者对应的 `pyenv local`）激活环境，然后再启动 Python。

2. **动态寻找解析器路径 (推荐)**：
   不再硬编码绝对路径，让脚本自己去找兄弟目录的虚拟环境，例如：
   ```python
   # 自动寻找 MediaCrawler 目录下是否自带 .venv 虚拟环境
   venv_python = os.path.join(crawler_dir, ".venv", "bin", "python")
   
   # 如果找不到自带的，再降级使用运行当前脚本的 python 或系统 python
   python_executable = venv_python if os.path.exists(venv_python) else sys.executable
   ```
   这就要求你在分发给别人前，先在对方机器的 `MediaCrawler` 里面 `python -m venv .venv && pip install -r requirements.txt` 创建好统一结构。

3. **终极发版形态：Docker (容器化)**：
   如果跨机器分发过于痛苦，可以写一个 `Dockerfile`，把整个环境、爬虫代码、截图代码、依赖库和底层二进制依赖揉在一块，一键打包成镜像发放，真正做到“一份镜像，随处运行”。
