# 视频截图模块 (VideoScreenshotter) — 交接文档

> 本文档面向后续开发者，说明视频截图模块的功能、接口定义、数据落盘行为和高级调试选项。

---

## 1. 模块功能

这是一个高内聚、零外部服务依赖的独立本地视频抽帧处理模块。该模块将从数据库中获取已判定为**合格且下载成功**的视频，均匀抽帧并结合图像相似度算法进行去重，最终留下具有代表性的截帧图片集。

**核心能力：**
- **均匀抽帧**：基于 OpenCV，从视频的开始至结束，完全平分为 `N` 个时间点抽取画面。
- **本地感知哈希去重**：内置基于 aHash (均值哈希算法) 算法。通过对比相邻留存画面的**汉明距离** (Hamming Distance)，智能丢弃高度重复或相似的连续画面。
- **无缝衔接数据库**：根据约定的 SQLite 状态码自动读取待处理列表，并将生成好的本地图片绝对路径用 `,` 拼接回写至数据库，供下游模块（如 OCR）无脑提取。
- **可视化调试模式**：内置独有的 `--debug` 模式，便于直观观察所设定的哈希阈值是否出现过度裁剪或保留过多废片的情况。

## 2. 启动方式

### 方式 A：作为普通模块调用 `run()` 接口（推荐流水线与 WebUI 使用）

该模块暴露的主入口函数位于 `screenshot_module.py` 中：

```python
from VideoScreenshotter.screenshot_module import run

# 执行该函数将自动寻找所有满足条件且从未被抽帧过的视频进行处理
processed_count = run(
    task_id="20260228-4YZJ",  # 指定流水线批次处理（可选参数）
    item_ids=["bilibili_BV1xx"], # 若只针对某单条/某几条视频进行重试，传入 item_id 数组（可选参数）
    num_frames=40,               # [高级参数] 期望将视频等分为多少个切片进行抽取。默认：40。
    dedup_threshold=5,           # [高级参数] aHash 的汉明距离阈值。默认：5。
                                 # (值越大：判断为相同的容忍度越高，剩余图片越少；值越小：判断越严格，保留细节变多，剩余图片越多)
    debug=False                  # 是否开启调试模式，分别保存两份对照图片。
)
```

### 方式 B：通过命令行运行（适合单独模块调试测试）

```bash
cd VideoScreenshotter
# 处理全部遗留的新视频，启用调试对照模式，指定抽取50帧且阈值提高到8
python screenshot_module.py --num_frames 50 --threshold 8 --debug
```

## 3. 数据库上下游流转与依赖关系

此模块与上游（爬虫下载）和下游（OCR识别）的全部数据交换，**皆通过项目根目录的唯一的 `MediaCrawler/data/media_items.db` 完成**。

### 模块读取 (Input) 定位条件
模块执行时，会到 `media_items` 表里查询复合以下特征的数据进入待处理队列：
- `content_type = 'video'`（仅针对于视频项）
- `initial_passed = 1`（初筛阶段合格，意味着内容具有分析价值）
- `media_downloaded = 1`（已经成功下载落盘到本地的真视频文件）
- **`video_screenshots IS NULL` 或者 `video_screenshots = ''`**（防止重复针对同一个视频多次盲目截图）

### 模块写入 (Output) 结果
每抽取完成一部视频，模块会将其去重保留下来的那些**图片的绝对路径**使用英文逗号 `,` 串联成一长串字符串。

**回写目标字段：** 更新到当前条目对应的 `video_screenshots` 列中。

*(下游 OCR 模块只需去数据库寻找 `initial_passed=1` 并且拥有 `video_screenshots` 不为空的列，将其依逗号 `.split(',')` 劈开，便能得到全部图集以供识别。)*

## 4. 关键文件与环境架构清单

VideoScreenshotter 是一个**完全独立**的模块文件夹，拥有自己的逻辑内脏，不混杂在爬虫目录内。

| 目录/文件 | 职责 |
|------|------|
| `screenshot_module.py` | 主入口。包含 argparse 接收和 `run()` 函数，负责协调。 |
| `core/image_algo.py` | 纯净的图像比对算法核心库，产出哈希值与度量差异距离。 |
| `core/screencap.py` | 基于 OpenCV 进行视频抽帧和执行比对存储的主体逻辑。 |
| `store/db_helper.py` | 跨模块查询 `MediaCrawler` 目录里的源库表并更新写入新列。 |
| `data/screenshots/` | 所有抽出来的截图的默认收容落盘仓库。 |

## 5. 调试注意事项 (Debug 模式详述)

由于每类视频（比如舞蹈视频画面变动巨大，相亲静态图视频几乎没变动）所需的最佳排重灵敏度可能不同，模块特别提供了 `--debug` 或 `debug=True`。

**普通模式：**
所有经过筛选**确认留存**的对应视频帧会被直接放在所属平台的子目录下，如：
`VideoScreenshotter/data/screenshots/bilibili/116147.../` （里面是一堆留下的图片）

**调试模式下落盘结构的异变：**
当您开启了 debug，针对每一个正在处理的视频，它不光会把最终保留的结果落盘，为了对照，它会在所属 id 的文件夹内**额外创建两个子目录**：

- **`/原始截图/`**: 无条件存放那 40 张纯正抽出的所有画面。
- **`/去重后图片/`**: 存放经过算法裁切筛选后真正幸存的那一部分碎片。

> [!TIP]
> **排错建议**：如果您利用调试模式发现“原始图很多不同”，导致“合并后依然太多（大于10张）”，说明算法判断过于严格，建议调高 `--threshold` (如改为 8) 使其过滤变强；反之如果发现误杀了重要画面，建议降低至 3 以下。

## 6. 日志规范与自动归档

为了严格遵守《系统架构设计》的要求，截图模块配备了标准化的分级长效日志系统。

**落盘位置：**
所有日志自动输出并保存至本模块目录下的：`VideoScreenshotter/logs/screenshotter.log`

**日志内容格式规范：**
`[精确时间] [Task_ID] [VideoScreenshotter] [INFO/ERROR] - 具体日志内容`
通过携带统一下发的 `task_id`，开发人员可以轻松地在统一存储池中，将爬虫、截图等多个孤立模块但在同一次流水线任务中产生的日志完全串联起来。

**防爆盘设计 (Daily Log Rotation)：**
程序内部使用了按天滚动的安全设计。每天午夜 00:00:00，系统会自动将写满一整天的 `screenshotter.log` 切割打包存为历史冷冻文件（如 `screenshotter.log.2026-03-02`），并创建一个新的同名空文件继续记录新一天的运行情况。
系统默认最多**只保留最近 7 天**的历史日志，超过 7 天的文件将会被自动销毁，彻底免除长期运行导致的服务器磁盘占满隐患。
