# 采存一体化爬虫模块 — 交接文档

> 本文档面向后续开发者，说明爬虫模块的功能、接口、数据结构和注意事项。

---

## 1. 模块功能

基于 MediaCrawler 开源项目魔改，支持 **小红书 (XHS)** 和 **B站 (Bilibili)** 两个平台的关键词搜索爬取。核心能力：

- 按关键词搜索，按时间倒序翻页爬取
- 时间窗口过滤（只保留指定时间段内的内容）
- 视频大小拦截（超限不下载，但基础信息仍入库）
- 所有内容统一写入 SQLite，通过初筛和未通过的都会记录
- 每次运行带有唯一 `task_id`，贯穿数据库和日志

---

## 2. 启动方式

### 方式 A：通过 `run()` 函数（推荐，供调度模块/调试网站调用）

```python
from crawler_module import run

task_id = run(
    platform="bili",              # "xhs" 或 "bili" (注意小红书会自动强制开启 CDP 真实浏览器模式以绕过验证码风控，B站使用普通的无头模式)
    keywords="bfb,大数据求偶",     # 多关键词用英文逗号分隔
    time_range_start_s=1772238600, # Unix 秒级时间戳，0=不限
    time_range_end_s=1772281800,
    max_notes=50,                  # 每个关键词的有效爬取上限
    max_video_size_mb=50,          # 视频大小上限(MB)，0=不限
    headless=True,                 # 无头模式 (针对非 CDP 模式有效)
)
# 返回 task_id，如 "20260228-QPP2"
```

### 方式 B：通过命令行（原有方式，仍可用）

```bash
python main.py --platform bili --keywords "bfb"
```

## 3. SQLite 数据库

**位置**：`data/media_items.db`

**表结构 `media_items`**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `item_id` | TEXT | 平台内唯一 ID（XHS note_id / B站 aid） |
| `platform` | TEXT | `"xhs"` 或 `"bilibili"` |
| `task_id` | TEXT | 批次号，如 `20260228-QPP2` |
| `content_type` | TEXT | `"normal"` / `"video"` |
| `title` | TEXT | 标题 |
| `desc` | TEXT | 描述正文 |
| `publish_time` | INTEGER | 发布时间（Unix 秒） |
| `user_id` | TEXT | 作者 ID |
| `nickname` | TEXT | 作者昵称 |
| `liked_count` | INTEGER | 点赞数 |
| `comment_count` | INTEGER | 评论数 |
| `share_count` | INTEGER | 分享数 |
| `collected_count` | INTEGER | 收藏数 |
| `play_count` | INTEGER | 播放数（仅B站，XHS为0） |
| `coin_count` | INTEGER | 投币数（仅B站） |
| `danmaku_count` | INTEGER | 弹幕数（仅B站） |
| `content_url` | TEXT | 原始链接 |
| `video_url` | TEXT | 视频流地址 |
| `image_urls` | TEXT | 图片URL列表（逗号分隔） |
| `tag_list` | TEXT | 标签（逗号分隔） |
| `source_keyword` | TEXT | 搜索关键词 |
| `initial_passed` | INTEGER | 1=通过初筛，0=被拦截 |
| `filter_reason` | TEXT | 拦截原因：`time_older` / `time_newer` / `NULL` |
| `media_downloaded` | INTEGER | 1=图片/视频已成功下载到本地存储，0=未下载 |
| `local_media_paths` | TEXT | 下载到本地的实际图片/视频相对路径（如 `data/bili/videos/116148.../video.mp4`） |
| `created_at` | INTEGER | 记录创建时间 |

**唯一约束**：`UNIQUE(platform, item_id, task_id)`

**常用查询示例**：
```sql
-- 查看某次任务的结果概况
SELECT initial_passed, filter_reason, COUNT(*)
FROM media_items WHERE task_id = '20260228-QPP2'
GROUP BY initial_passed, filter_reason;

-- 查看某次任务中通过初筛的内容
SELECT title, publish_time, source_keyword
FROM media_items WHERE task_id = '20260228-QPP2' AND initial_passed = 1;
```

## 4. 日志格式

```
[时间] [task_id] [模块名] [级别] (文件:行号) - 消息
```

示例：
```
[2026-02-28 20:30:44] [20260228-QPP2] [MediaCrawler] [INFO] (core.py:662) - [BilibiliCrawler.get_bilibili_video] Sleeping for 2 seconds...
```

可按 `task_id` 快速过滤某次运行的所有日志：
```bash
grep "20260228-QPP2" logs/xxx.log
```

## 6. 平台差异与注意事项

### 小红书 (XHS)
- 搜索关键词自动加 `#` 前缀以过滤广告笔记
- 时间排序非严格连续，采用"连续3条旧帖才熔断"策略
- 详情 API 偶有失败，不影响整体流程
- **反爬/风控说明**：小红书存在极强的翻页风控（通常在爬取10~20条后返回异常数据），因此 `run()` 方法中已硬编码 `config.ENABLE_CDP_MODE = True`。爬取时会自动接管本机的 Chrome/Edge 的真实运行进程模拟真实人类环境绕过封禁。如果遇到登录失效打印 `waiting for scan code login`，需前往刚拉起的 Chromium 窗口处扫码登录。

### B站 (Bilibili)
- 时间排序严格，遇到第一条旧帖即熔断
- 视频下载前通过 HEAD 请求检查大小
- 并发数限制为 2（`MAX_CONCURRENCY_NUM`），避免 403

### 通用
- JSON 存储仍在并行输出，可直接查看 `data/{platform}/json/` 下的文件
- 媒体文件按 `item_id` 建立子文件夹：`data/{platform}/images/{id}/` 或 `data/{platform}/videos/{id}/`
- 同一条目被多次爬取时，媒体文件会覆盖（内容相同无影响），SQLite 按 `(platform, item_id, task_id)` 去重
- **媒体文件下载状态**：一旦图文或视频下载成功，系统会同步更新 `media_items` 表里的 `media_downloaded = 1` 并填充 `local_media_paths`。断点或崩溃时可通过数据库判断哪些文件已成功落盘。

## 7. 下游模块对接指南

下游模块（截图、OCR、最终筛选等）只需要：

1. **读取 SQLite**：查询 `WHERE task_id = ? AND initial_passed = 1 AND media_downloaded = 1` 获取本批次通过初筛**且**媒体资源已确保安全落盘的笔记数据。
2. **提取本地文件**：从记录的 `local_media_paths` 字段直接提取文件绝对路径使用，无需再手动拼接 `data/{platform}/...` 以防出入。
3. **写回结果**：将处理结果（OCR 文本、最终筛选结论等）更新到 SQLite 的对应新列中。

如需扩展 `media_items` 表结构，在 `store/unified_store.py` 的建表 SQL 中添加列即可。
